# INDIGO Constants Extraction

## Overview

The `src/constants.rs` file contains all INDIGO protocol constants automatically extracted from the upstream INDIGO C library. These constants provide type-safe access to standard INDIGO property and item names.

## Automatic Generation

Constants are automatically extracted during the build process by [`build.rs`](../build.rs). The extraction happens whenever:

1. The project is built from a clean state (`cargo clean && cargo build`)
2. The INDIGO submodule exists at `sys/externals/indigo`

The build script parses `sys/externals/indigo/indigo_libs/indigo/indigo_names.h` and extracts all `#define` statements that end with `_NAME`, converting them to Rust constants.

### Example Extraction

C header definition:

```c
#define CONNECTION_PROPERTY_NAME "CONNECTION"
#define INFO_PROPERTY_NAME "INFO"
```

Generated Rust constants:

```rust
pub const CONNECTION_PROPERTY: &str = "CONNECTION";
pub const INFO_PROPERTY: &str = "INFO";
```

## Updating Constants

When the upstream INDIGO library adds new properties or changes existing ones, you need to regenerate the constants file.

### Method 1: Using the Update Script (Recommended)

```bash
./scripts/update_constants.sh
```

This script will:

1. Update the INDIGO git submodule to the latest version
2. Clean build artifacts
3. Trigger constant regeneration
4. Verify the constants were generated successfully

### Method 2: Manual Update

```bash
# Update the INDIGO submodule
git submodule update --remote sys/externals/indigo

# Clean and rebuild to trigger extraction
cargo clean
cargo build
```

### Method 3: Update to Specific INDIGO Version

```bash
# Navigate to the submodule
cd sys/externals/indigo

# Checkout specific version
git fetch --tags
git checkout v2.0.300  # or any other version/tag

# Return to project root
cd ../../..

# Regenerate constants
cargo clean && cargo build
```

## Usage

Constants are available through the `libindigo::name` module:

```rust
use libindigo::name;

// Access standard property names
assert_eq!(name::CONNECTION_PROPERTY, "CONNECTION");
assert_eq!(name::INFO_PROPERTY, "INFO");

// Access property item names
assert_eq!(name::CONNECTION_CONNECTED_ITEM, "CONNECTED");
assert_eq!(name::INFO_DEVICE_NAME_ITEM, "DEVICE_NAME");
```

## Implementation Details

### Build Script Logic

The [`build.rs`](../build.rs) script:

1. **Checks for INDIGO submodule**: Looks for `sys/externals/indigo`
2. **Parses header file**: Reads `indigo_names.h` line by line
3. **Extracts constants**: Uses regex to match `#define CONSTANT_NAME "value"`
4. **Generates Rust code**: Writes constants to `src/constants.rs`
5. **Adds metadata**: Includes header comment with generation instructions

### Regex Pattern

```rust
r#"^#define\s+(?<name>\w+)_NAME\s+"(?<value>.+)"\s*$"#
```

This pattern matches:

- `#define` at the start of a line
- Constant name ending in `_NAME`
- String value in quotes

### Generated File Format

```rust
// This file is automatically generated by build.rs
// DO NOT EDIT MANUALLY
//
// To regenerate this file:
//   1. Update the INDIGO submodule: git submodule update --remote sys/externals/indigo
//   2. Run: cargo clean && cargo build
//
// Source: sys/externals/indigo/indigo_libs/indigo/indigo_names.h

pub const CONSTANT_NAME: &str = "VALUE";
// ... more constants ...
```

## Troubleshooting

### Constants Not Updating

If constants don't update after running the build:

1. **Ensure submodule exists**:

   ```bash
   ls -la sys/externals/indigo/indigo_libs/indigo/indigo_names.h
   ```

2. **Check build output** for warnings about constant extraction

3. **Force regeneration**:

   ```bash
   rm src/constants.rs
   cargo clean
   cargo build
   ```

### Submodule Not Found

If the INDIGO submodule is missing:

```bash
git submodule update --init --recursive sys/externals/indigo
```

### Build Errors

If you encounter build errors related to constants:

1. Check that `src/constants.rs` exists and is valid Rust code
2. Verify the INDIGO submodule is at a compatible version
3. Try regenerating from a clean state

## Version Control

The `src/constants.rs` file **is committed to version control**. This ensures:

- Builds work without requiring the INDIGO submodule
- Constants remain stable between INDIGO updates
- Changes to constants are tracked in git history

When updating constants:

1. Review changes: `git diff src/constants.rs`
2. Verify all tests pass: `cargo test`
3. Commit the updated file: `git add src/constants.rs && git commit -m "Update INDIGO constants to version X.Y.Z"`

## Migration from props.rs

The old `props.rs` file (with `_NAME` suffix constants) has been removed in favor of the automatically generated `src/constants.rs`. The new constants:

- Have cleaner names (no `_NAME` suffix)
- Are automatically synchronized with INDIGO
- Include generation metadata
- Are documented with their source

No code changes are required as the constants are accessed through the same `libindigo::name` module.
