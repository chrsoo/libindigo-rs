# Phase 4 Refactoring: libindigo-sys - Pure Bindgen Crate

## Completed: 2026-03-06

This document summarizes the Phase 4 refactoring that transformed `libindigo-sys` into a pure bindgen FFI bindings crate.

## Changes Made

### 1. Cleaned up `sys/src/lib.rs`

**Before:** 475 lines with extensive higher-level Rust abstractions:

- Custom implementations for `indigo_property::new()`, `indigo_item::new()`, `indigo_item::text()`, `indigo_item::number()`
- Helper functions: `buf_to_str()`, `str_to_buf()`, `ptr_to_str()`, `copy_from_str()`
- Trait implementations: `PartialEq`, `Hash`, `Eq`, `Sync`, `Send` for `indigo_client`
- `FamHeader` trait implementation for `indigo_property`
- `Default` implementation for `indigo_item`
- Test code with client callbacks (474 lines of test code)
- Manual constants (`DEFAULT_PORT`, `DEFAULT_HOST`, `BUF_SIZE`, etc.)

**After:** 8 lines - pure bindgen output:

```rust
#![allow(non_upper_case_globals)]
#![allow(non_camel_case_types)]
#![allow(non_snake_case)]
#![allow(unused_imports)]
#![allow(dead_code)]

// Raw bindgen-generated FFI bindings to the INDIGO C library
include!(concat!(env!("OUT_DIR"), "/bindings.rs"));
```

### 2. Cleaned up `sys/Cargo.toml`

**Removed runtime dependencies:**

- `log = "0.4"` - Not needed for raw FFI bindings
- `fambox = "0.2.0"` - Used for flexible array members (higher-level abstraction)
- `enum_primitive = "0.1.1"` - Used for enum conversions (higher-level abstraction)

**Removed build dependencies:**

- `log = "0.4"` - Not needed
- `test-log = "0.2.16"` - Not needed
- `env_logger = "*"` - Not needed

**Kept only essential build dependencies:**

- `bindgen` - Required for generating FFI bindings
- `semver` - Used for version parsing
- `regex` - Used for parsing INDIGO headers
- `rev_buf_reader` - Used for reading build logs

### 3. Verified `sys/build.rs`

The `sys/build.rs` file is correctly configured:

- ✅ Compiles INDIGO C library using `make`
- ✅ Generates raw FFI bindings using `bindgen`
- ✅ No higher-level code generation beyond bindgen
- ✅ Proper callbacks for renaming bindgen types (e.g., `indigo_value`, `indigo_text`)

### 4. Root `build.rs` Analysis

The root `build.rs` file generates:

- `name.rs` - INDIGO protocol constants (property/item names)
- `interface.rs` - Device interface enum bindings

**Decision:** Keep root `build.rs` as-is. It generates constants for the main `libindigo` crate, not for `libindigo-sys`. This is appropriate separation of concerns:

- `libindigo-sys` = Raw C bindings only
- `libindigo` (main crate) = Higher-level API with constants and conveniences

The refactored `libindigo-sys` now fully complies with the Phase 4 requirements:

### ✅ Pure FFI Bindings Crate

- Contains ONLY bindgen-generated raw C bindings
- No Rust-level abstractions or convenience functions
- No higher-level wrappers

### ✅ Minimal Dependencies

- Zero runtime dependencies
- Only essential build dependencies for bindgen and C compilation

### ✅ Proper Separation of Concerns

- `libindigo-sys` - Raw FFI bindings (this crate)
- `libindigo-ffi` - Higher-level FFI wrappers and abstractions

## Migration Notes

### For Code That Used Higher-Level Abstractions

The removed helper functions and implementations should be moved to `libindigo-ffi` if needed:

**Removed from `libindigo-sys`:**

- `indigo_property::new()` → Move to `libindigo-ffi`
- `indigo_item::new()`, `::text()`, `::number()` → Move to `libindigo-ffi`
- `buf_to_str()`, `str_to_buf()`, `ptr_to_str()` → Move to `libindigo-ffi`
- Trait implementations (PartialEq, Hash, etc.) → Move to `libindigo-ffi`

**Constants:**

- Manual constants like `DEFAULT_PORT`, `DEFAULT_HOST` → Move to `libindigo` or `libindigo-ffi`
- INDIGO name constants are already in `libindigo::name` module (generated by root `build.rs`)
